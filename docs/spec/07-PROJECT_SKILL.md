# hvacptcharts authoring rules

The previous WordPress site shipped with three structural failures:

1. **Fabricated PT data.** Every saturation pressure on the site was an LLM hallucination wrong by 2-15×. Several values exceeded the refrigerant's critical pressure (physically impossible).
2. **Template-swap copy.** The detail page template hardcoded R-410A-specific prose with the refrigerant name templated in. Every refrigerant page told the reader that refrigerant is a 50/50 blend of R-32 and R-125. R-22 page said R-22 is. R-290 (propane) said propane is. R-717 (ammonia) said ammonia is.
3. **Wrong safety classifications.** Every refrigerant inherited "A1 non-flammable" from the template. In reality R-32 is A2L, R-290 is A3, R-717 is B2L. Telling installers that flammable refrigerants are non-flammable is a safety-critical error.

The rebuild's structural defenses against each:

| Failure | Structural defense in this codebase |
|---|---|
| Fabricated PT | All PT data is generated by `scripts/generate-refrigerant-data.py` from CoolProp + manufacturer datasheets. Committed to `data/refrigerants.json`. Validated at build time against 5 known-good anchor values. |
| Template-swap copy | Templates contain zero refrigerant-specific prose. All per-refrigerant copy is in `content/refrigerants/{slug}.mdx`. Templates render data + MDX; never narrate. |
| Wrong safety class | `Refrigerant.safetyClass` is a Zod-validated enum. Templates render the value from the data layer, not from prose. |

## Hard rules

These are absolute. If you cannot comply, stop and ask Marko.

### Rule 1: No fabricated PT values, ever

**Forbidden:**
- Writing a saturation pressure number directly into a `.tsx`, `.ts`, `.mdx`, or `.md` file.
- "Reasonable estimate" or "approximate" PT values when the actual data is missing.
- Computing a PT value by analogy ("R-32 is similar to R-410A so it should be about...").
- Inventing a PT chart for a refrigerant not yet in `data/refrigerants.json`.

**Required:**
- All PT values come from `getRefrigerant(slug)` or `getPressureAtTempF(slug, tempF)` at runtime.
- For build-time display in templates: import from `@/data/refrigerants` and reference fields.
- For prose in MDX files: pressure values are rendered via React components that pull from the data layer (e.g., `<PressureAtTemp slug="r-410a" tempF={70} />`), not typed as literal numbers.
- For new refrigerants: add an entry to `data/refrigerants.config.json` and run `pnpm run generate-data` to populate the PT chart from CoolProp or a verified manufacturer source.

**Self-check before writing any number that looks like a pressure or temperature:**
- Did this number come from `getRefrigerant()` or a CoolProp computation?
- If I am typing this number into a file, why is it not coming from the data layer?
- If the answer is "I'm copying it from a manufacturer datasheet for a manual-entry blend," is the source cited in `data/manufacturer-blends/{slug}.json`?

### Rule 2: No template-swap copy

**Forbidden:**
- Writing prose in a template that describes the refrigerant being rendered. Even if it uses `{r.displayName}` interpolation.
- Statements about composition, lubricant, application, safety, GWP, or properties in shared template files.
- Reusing copy from one MDX file in another by changing the refrigerant name.

**Required:**
- All refrigerant-specific narrative lives in `content/refrigerants/{slug}.mdx`.
- The template (`src/app/refrigerant/[slug]/page.tsx`) renders structural elements (hero, table, chart, footer) and reads structured fields from the data layer, but never embeds prose about the specific refrigerant.
- Composition is rendered from `r.composition` array (not narrated in prose).
- Safety class is rendered via `<SafetyClassChip safetyClass={r.safetyClass} />` (not narrated).
- Lubricant compatibility is rendered from `r.lubricants.compatible` array (not narrated).
- GWP is rendered from `r.environmental.gwp100Ar5` (not narrated in template).

**Self-check before writing prose in any `.tsx` file under `src/app/refrigerant/`:**
- Could a reader of this template tell which specific refrigerant the page is about, from the template file alone?
- If yes, that prose belongs in MDX. Move it.

### Rule 3: Safety class is structural, never prose

**Forbidden:**
- "{refrigerant} is non-flammable" or "{refrigerant} is A1" written as prose in a template or MDX, when the data layer disagrees.
- Hardcoding any safety classification value.
- Lumping A1 and A2L together as "safe" or A2L and A3 together as "flammable."

**Required:**
- Every refrigerant page renders `<SafetyClassChip safetyClass={r.safetyClass} variant="card" />` in the hero.
- Card variant displays: the class code (A1/A2L/A3/B1/B2L/B2/B3), the descriptive label ("Non-flammable" / "Mildly flammable" / "Highly flammable" / "Toxic"), and the standard interpretation.
- In MDX prose, if a refrigerant's flammability is discussed, the prose must match the safety class in the data layer. CI runs a check.

**Class code reference (these are the only valid values):**
- A1 — Lower toxicity, non-flammable
- A2L — Lower toxicity, lower flammability (mildly flammable, low burning velocity)
- A2 — Lower toxicity, flammable
- A3 — Lower toxicity, higher flammability (e.g. propane, isobutane)
- B1 — Higher toxicity, non-flammable
- B2L — Higher toxicity, lower flammability (e.g. ammonia)
- B2 — Higher toxicity, flammable
- B3 — Higher toxicity, higher flammability

### Rule 4: Every factual claim has a source

**Forbidden:**
- Stating a GWP value without sourcing IPCC AR5 or AR6.
- Stating a regulatory date (phase-down milestone, production ban) without linking to the EPA / Montreal Protocol / Kigali source.
- Stating a property value (critical temp, boiling point, molar mass) without sourcing CoolProp or ASHRAE 34.
- Stating an industry practice or recommendation ("target superheat of 10-15°F") without citing ACCA, manufacturer literature, or ASHRAE.

**Required:**
- Sources are registered once in `data/sources.json` and referenced by ID.
- In MDX, citations are added as superscript footnote markers via the `<Cite />` component: `<Cite id="ipcc-ar5-ch8a1" />`.
- Provenance footer on every page lists the data sources for that page.

**Self-check before writing any factual sentence:**
- Could this be wrong?
- If yes, where did I get it from?
- If I don't know, find a source or remove the sentence.

### Rule 5: Verify CoolProp output against anchors

The verify script (`scripts/verify-refrigerant-data.ts`) runs at build. It checks the regenerated PT data against 5 known-good anchor values. If any drift more than ±5%, build fails.

```typescript
// Anchor values verified against Arkema Forane PT charts and ASHRAE Handbook of Refrigeration 2022.
const ANCHORS: Array<{ slug: string; tempF: number; expectedPsig: number; tolerance: number }> = [
  { slug: 'r-22',   tempF: 70, expectedPsig: 121.4, tolerance: 0.05 },
  { slug: 'r-410a', tempF: 70, expectedPsig: 201.5, tolerance: 0.05 },
  { slug: 'r-134a', tempF: 70, expectedPsig: 71.1,  tolerance: 0.05 },
  { slug: 'r-32',   tempF: 70, expectedPsig: 188.5, tolerance: 0.05 },
  { slug: 'r-404a', tempF: 70, expectedPsig: 165.3, tolerance: 0.05 },
];
```

If you ever find yourself editing `data/refrigerants.json` directly to make this pass, **stop**. The fix belongs in `refrigerants.config.json` or `manufacturer-blends/`, and the data should be regenerated. Direct edits to the generated file are a code smell.

### Rule 6: No paragraphs longer than 3 sentences

Marko's standing rule across the portfolio. Strictly enforced in this rebuild.

**Forbidden:**
- Prose paragraphs with 4+ sentences in MDX content files.
- Wall-of-text sections in template content.

**Required:**
- Break dense content into short paragraphs, lists, tables, or callouts.
- Each paragraph carries one idea.

### Rule 7: No `localStorage`, no client-side data fetching for PT data

PT data is static and shipped at build time. No reason to fetch it at runtime.

**Forbidden:**
- `fetch()` calls to refrigerant data from client components.
- `localStorage` or `sessionStorage` for caching refrigerant data.
- Lazy-loading the PT data via dynamic import (unless the page genuinely needs only one refrigerant — and even then, prefer prop-passing from a server component).

**Required:**
- Refrigerant data imported from `@/data/refrigerants` at build time.
- Server components do the data lookup; client components receive the data as props.

### Rule 8: Do not invent refrigerants

The site covers 61 specific refrigerants. The list is fixed by the existing site's URL structure (which we preserve). Do not add refrigerants that aren't in the list. Do not generate pages for hypothetical or proposed refrigerants.

**If a real new refrigerant comes onto the market that should be added** (e.g. a freshly-released HFO blend), it's an explicit Marko-approved change: add to the config, regenerate data, write MDX, deploy.

### Rule 9: Verify mathematical formulas against named sources

Calculator math (superheat, subcooling, charge calculations, retrofit compatibility logic) is referenced from:

- ACCA Manual J (load calculation) — full implementation deferred per Marko
- ACCA Manual S (equipment selection)
- ACCA Manual D (duct design)
- ACCA Manual S/T (charging procedures)
- ASHRAE Handbook of Fundamentals (psychrometric, properties)
- Manufacturer service literature for system-specific charging tables

**Forbidden:**
- Inventing or "deriving from first principles" any formula that has a standard reference.
- Using a formula from a forum or stack-overflow answer.

**Required:**
- Formula sources cited in code comments and in the calculator page's "underlying math" section.

### Rule 10: No emoji in production content

Decorative emojis (📊 🔧 💰 ❄️ 🌡️) are not used. Lucide React icons via the icon system. Exception: Marko's choice, where a relevant emoji like ⚠️ matches a callout's actual semantic content.

## Companion file: `verify.ts`

```typescript
// .claude/skills/hvacptcharts/verify.ts
// Optional companion module used by build. The skill itself is documentation;
// this file enforces a subset of the rules at compile time.

import { z } from 'zod';
import anchors from './reference/coolprop-anchors.json';
import { Refrigerant } from '@/data/refrigerants';

const Anchor = z.object({
  slug: z.string(),
  tempF: z.number(),
  expectedPsig: z.number(),
  tolerance: z.number(),
});

const AnchorList = z.array(Anchor);

export function verifyAgainstAnchors(refrigerants: Refrigerant[]): { ok: boolean; errors: string[] } {
  const parsedAnchors = AnchorList.parse(anchors);
  const errors: string[] = [];

  for (const anchor of parsedAnchors) {
    const r = refrigerants.find(x => x.slug === anchor.slug);
    if (!r) {
      errors.push(`Anchor refrigerant ${anchor.slug} not found in dataset.`);
      continue;
    }
    const point = r.ptChart.find(p => p.tempF === anchor.tempF);
    if (!point) {
      errors.push(`${anchor.slug}: no PT point at ${anchor.tempF}°F.`);
      continue;
    }
    const actual = point.bubblePsig;
    const expected = anchor.expectedPsig;
    const diff = Math.abs(actual - expected) / expected;
    if (diff > anchor.tolerance) {
      errors.push(
        `${anchor.slug} at ${anchor.tempF}°F: expected ~${expected} PSIG, got ${actual} PSIG ` +
        `(${(diff * 100).toFixed(1)}% off, tolerance ±${(anchor.tolerance * 100).toFixed(0)}%)`
      );
    }
  }

  // Also verify physical impossibilities
  for (const r of refrigerants) {
    if (r.physical.critical.pressurePsig === null) continue;
    for (const p of r.ptChart) {
      if (p.bubblePsig > r.physical.critical.pressurePsig) {
        errors.push(
          `${r.slug}: saturation pressure ${p.bubblePsig} PSIG at ${p.tempF}°F ` +
          `exceeds critical pressure ${r.physical.critical.pressurePsig} PSIG ` +
          `(physically impossible).`
        );
      }
    }
  }

  return { ok: errors.length === 0, errors };
}
```

## Wiring this into the build

`package.json`:

```json
{
  "scripts": {
    "generate-data": "python scripts/generate-refrigerant-data.py",
    "verify-data": "tsx scripts/run-verify.ts",
    "prebuild": "pnpm run verify-data",
    "build": "next build"
  }
}
```

`scripts/run-verify.ts`:

```typescript
import { refrigerants } from '@/data/refrigerants';
import { verifyAgainstAnchors } from '../.claude/skills/hvacptcharts/verify';

const { ok, errors } = verifyAgainstAnchors(refrigerants);
if (!ok) {
  console.error('Refrigerant data verification failed:');
  errors.forEach(e => console.error('  - ' + e));
  process.exit(1);
}
console.log(`✓ Verified ${refrigerants.length} refrigerants against ${10} anchors. All within tolerance.`);
```

A failed `pnpm build` is the structural last line of defense.

## How CC actually uses this skill

When Claude Code is given a prompt like:

> "Add the new R-454C refrigerant page"

The system prompt for that CC session includes:
- The repository's `CLAUDE.md` (top-level instructions)
- Any `.claude/skills/*/SKILL.md` files that match the trigger keywords

This skill matches the trigger "refrigerant page" → loaded.

CC reads the rules, sees:
- Rule 1 forbids writing PT values into TSX files → CC reads from data layer.
- Rule 2 forbids prose in templates → CC creates `content/refrigerants/r-454c.mdx`.
- Rule 3 enforces safety class as data → CC uses `<SafetyClassChip>` from `r.safetyClass`.
- Rule 4 requires sources for every claim → CC adds citations.
- Rule 5 references the anchor list → CC ensures the new data passes verify.

The skill is the contract between Marko's quality bar and CC's defaults. It is the difference between "CC tries to be careful" and "CC structurally cannot ship the failures the old site had."

## Maintenance

Update the skill (this file at `.claude/skills/hvacptcharts/SKILL.md`) when:

- A new structural failure mode is discovered (add a rule).
- A new refrigerant type is added (update the safety class reference if needed).
- The anchor list needs expansion or correction.
- The build verification needs new checks.

Never remove a rule because it's "annoying." If a rule blocks legitimate work, the rule needs to be sharpened, not removed.
